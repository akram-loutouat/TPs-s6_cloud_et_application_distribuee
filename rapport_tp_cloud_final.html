Introduction

Ce rapport présente les travaux pratiques réalisés dans le cadre du module Cloud & Infrastructure. L'objectif est d'explorer, administrer et sécuriser deux hyperviseurs de type 1 : VMware ESXi et Proxmox VE, puis de les comparer selon des critères techniques et de cybersécurité.

Les travaux couvrent la prise en main des interfaces web, la configuration réseau, la création et gestion de machines virtuelles, l'application de mesures de durcissement, la mise en place d'un mini-lab isolé, et une analyse comparative argumentée.

Questions Partie A

Un attaquant obtient l'accès SSH à votre ESXi. Quels sont les risques ? Quelles mesures auraient pu l'en empêcher ?
Avec un accès SSH root à ESXi, l'attaquant contrôle totalement l'hyperviseur : il peut éteindre/supprimer toutes les VMs, exfiltrer leurs fichiers .vmdk (disques complets), modifier la configuration réseau pour intercepter le trafic, et effacer les logs locaux pour masquer ses traces. La persistance est triviale via ajout de clés SSH. Mesures préventives : désactiver SSH quand inutilisé, restreindre le port 22 par firewall, activer le lockdown mode avec vCenter, envoyer les logs vers un SIEM externe, timeout de session SSH configuré.
Pourquoi est-il dangereux de laisser le mode promiscuous activé sur un port group ?
Le mode promiscuous permet à une VM de capturer tout le trafic circulant sur le vSwitch, y compris celui destiné aux autres VMs. Cela constitue une vulnérabilité d'écoute passive (sniffing) : une VM compromise peut intercepter mots de passe, données métier, communications inter-VMs. Par défaut, ESXi règle ce paramètre sur Reject, garantissant l'isolation : chaque VM ne voit que son propre trafic.
Comment un SIEM peut-il exploiter les logs ESXi pour détecter une compromission ?
En configurant Syslog.global.logHost, les logs ESXi (hostd.log, auth.log, shell.log) sont envoyés en temps réel vers le SIEM. Celui-ci peut détecter des connexions SSH depuis des IP inconnues, des tentatives de brute-force, des modifications de configuration non planifiées, des arrêts de VMs hors maintenance, ou des pics d'activité inhabituels. L'avantage clé : si l'hôte est compromis, les logs distants sont intacts et permettent une investigation fiable.
Quel est l'intérêt de créer des comptes nominatifs plutôt que de partager le mot de passe root ?
Les comptes nominatifs garantissent la traçabilité individuelle : chaque action est attribuée à une personne précise dans les logs. Le partage du compte root rend l'audit impossible. De plus, les comptes nominatifs peuvent être limités à un rôle restreint (Read-Only) selon le principe du moindre privilège. En cas de départ d'un collaborateur, son accès est révoqué sans impacter les autres. Les politiques de mot de passe et d'expiration sont applicables individuellement.

RBAC Proxmox — avantage clé vs ESXi
Le RBAC Proxmox est natif et granulaire : accès par chemin (un seul VM, un pool), rôles personnalisés, groupes. ESXi standalone n'offre que Admin/Read-Only/No Access sans vCenter.

Questions Partie B

Quels avantages offre le RBAC de Proxmox par rapport à la gestion des comptes ESXi standalone ?
Le RBAC Proxmox permet une granularité que n'offre pas ESXi sans vCenter : définition de rôles personnalisés, attribution d'accès par chemin précis (ex: accès à une seule VM : /vms/100), gestion par groupes, realms multiples (PAM, PVE, LDAP). ESXi standalone ne propose que trois rôles fixes (Admin, Read-Only, No Access) appliqués à tout l'hôte. Le RBAC Proxmox permet donc d'appliquer réellement le principe du moindre privilège sans infrastructure vCenter coûteuse.
Pourquoi un conteneur LXC offre-t-il une isolation moindre qu'une VM ? Quel risque en cybersécurité ?
Un conteneur LXC partage le noyau Linux de l'hôte avec les autres conteneurs, contrairement à une VM KVM qui virtualise un noyau complet indépendant. En cas de CVE dans le noyau Linux ou d'évasion de conteneur (container escape), un attaquant peut potentiellement accéder à l'hôte Proxmox et à tous les autres conteneurs. Pour des services critiques ou sensibles (SOC, données personnelles, production), une VM KVM est préférable pour une isolation complète. Les LXC non privilégiés (unprivileged containers) réduisent ce risque mais ne l'éliminent pas totalement.
Pourquoi est-il critique de séparer le stockage des sauvegardes du stockage des VMs ?
Si sauvegardes et VMs partagent le même stockage, une corruption, une défaillance matérielle, ou un ransomware qui chiffre le datastore principal détruit simultanément les données ET les sauvegardes. La règle 3-2-1 stipule : 3 copies, sur 2 supports différents, dont 1 hors site. En stockant les sauvegardes vzdump sur un NFS externe ou un volume différent, on garantit la capacité de restauration même si le stockage principal est compromis.
Comment un attaquant pourrait-il exploiter une interface Proxmox exposée sur Internet ? Quelles protections mettre en place ?
Une interface Proxmox exposée est une cible critique : brute-force du compte root sur le port 8006, exploitation de CVE dans pveproxy, ou utilisation d'identifiants volés. L'accès root à Proxmox donne le contrôle total de toutes les VMs. Protections : ne jamais exposer le port 8006 sur Internet (utiliser un VPN pour l'accès distant), désactiver le login root direct (créer des comptes nominatifs), activer fail2ban, configurer le firewall Proxmox intégré, utiliser l'authentification à deux facteurs (TOTP supporté nativement), et surveiller les logs d'authentification via journald.

Synthèse sécurité
ESXi a une surface d'attaque naturellement plus faible (VMkernel minimal, pas d'OS hôte complet). Proxmox compense avec un RBAC natif complet, un firewall à 3 niveaux, et la puissance des outils Linux (fail2ban, AppArmor, iptables). En production sensible, les deux peuvent être durcis efficacement.

Exercice comparatif — Recommandation PME

📝
Scénario : PME 50 employés, 3 serveurs à virtualiser, budget limité, 1 admin IT à mi-temps, sauvegardes automatisées requises

Dans ce contexte, je recommande Proxmox VE pour les raisons suivantes :

Coût : Proxmox est entièrement gratuit (open source AGPLv3), ce qui est décisif pour une PME à budget limité. ESXi/vSphere nécessite des licences vCenter pour les fonctionnalités essentielles (RBAC complet, snapshots avancés, clustering), représentant plusieurs milliers d'euros.

Administration : L'interface Proxmox est complète et ne nécessite pas de composant supplémentaire (contrairement à vCenter). Un admin à mi-temps peut gérer l'ensemble depuis une seule interface web incluant réseau, stockage, sauvegardes et permissions.

Sauvegardes : vzdump intégré permet de planifier des sauvegardes automatiques de toutes les VMs avec compression et rétention configurable, sans logiciel tiers. Critique pour une PME sans équipe dédiée.

Sécurité : Le RBAC natif Proxmox permet de créer des comptes nominatifs avec rôles précis. Le firewall intégré (3 niveaux) offre une protection suffisante. Les mises à jour se font via apt, garantissant un cycle de correctifs régulier.

Réserve : Si la PME travaille avec des éditeurs certifiant uniquement sur VMware (SAP, Oracle...) ou nécessite un support contractuel solide, ESXi/vSphere reste justifiable malgré son coût.

Questions Partie C

Si vous deviez déployer un lab de forensic, laquelle des deux plateformes choisiriez-vous ? Justifiez.
Je choisirais Proxmox VE pour un lab de forensic. Les raisons principales : les snapshots rapides permettent de capturer l'état exact d'une VM compromise pour analyse sans la modifier (essential en forensic) ; le bridge vmbr1 sans uplink crée un réseau totalement isolé pour analyser des malwares sans risque de propagation ; les conteneurs LXC peuvent servir pour des analyses rapides et légères ; et la richesse des outils Linux disponibles (Volatility, Autopsy, tshark, sleuthkit) s'installe directement sur l'hôte Debian. De plus, vzdump permet de sauvegarder et transporter les images forensiques. Proxmox est gratuit et très utilisé dans les labs académiques de sécurité.
Quels sont les risques spécifiques liés au fait que Proxmox est un Debian complet ?
Étant basé sur Debian, Proxmox hérite d'une surface d'attaque plus large qu'ESXi : un SSH expose potentiellement tout l'hôte, des paquets inutiles installés augmentent le nombre de CVE exploitables, les dépendances apt peuvent introduire des bibliothèques vulnérables, et un attaquant avec accès root Linux peut contourner les protections de l'hyperviseur via l'OS sous-jacent. Des risques spécifiques incluent : privesc Linux vers l'hyperviseur, exploitation de bibliothèques Python/Perl utilisées par pvesh, et modifications directes des fichiers de configuration des VMs dans /etc/pve. La mitigation passe par un durcissement Linux classique : désactiver les services inutiles, fail2ban, AppArmor, minimiser les paquets installés.
Pourquoi Broadcom (anciennement VMware) a-t-il modifié ses licences ? Quel impact pour les PME ?
En 2024, Broadcom a racheté VMware et restructuré son modèle de licences : suppression des licences perpétuelles au profit d'abonnements, regroupement en bundles (VMware Cloud Foundation) forçant l'achat de fonctionnalités non nécessaires, et augmentation significative des prix. L'impact pour les PME est majeur : les coûts ont multiplié par 3 à 10 pour certains clients, rendant vSphere inabordable pour les petites structures. Cela a provoqué une migration massive vers des alternatives open source comme Proxmox VE, OpenStack, ou XCP-ng. Pour les PME, la décision de Broadcom a donc rendu Proxmox encore plus attractif comme alternative sérieuse et économique.

Questions Partie D

Le snapshot a permis de revenir en arrière. Mais que se passe-t-il si un attaquant a exfiltré des données avant la restauration ? Le snapshot protège-t-il contre cela ?
Non, le snapshot ne protège pas contre l'exfiltration de données. Il restaure l'état système (fichiers, configuration, mémoire) mais ne peut pas annuler les actions effectuées en dehors de la VM : si des données ont été copiées vers un serveur externe, ces copies persistent même après restauration. La restauration du snapshot efface les modifications sur la VM compromise mais pas les données déjà extraites. Pour contrer cela, il faut : une surveillance réseau (IDS/IPS capturant les flux suspects avant l'exfiltration), un SIEM analysant les transferts de données inhabituels, et une isolation réseau stricte limitant les destinations possibles des connexions sortantes.
Proposez 3 mesures supplémentaires de sécurité pour renforcer ce mini-lab.
1) Fail2ban sur VM-Server : détection automatique des tentatives de brute-force SSH (blocage après 5 échecs). 2) Désactivation des protocoles inutiles : désactiver IPv6 si non utilisé, limiter les services systemd démarrés au strict minimum (principe de surface minimale). 3) Audit avec auditd : installation du démon d'audit Linux pour journaliser toutes les commandes exécutées avec sudo, les modifications de fichiers critiques (/etc/passwd, /etc/ssh/sshd_config) et les connexions. Ces logs seraient envoyés vers un serveur de logs distant (logstash/rsyslog) pour protection contre leur suppression.
Comment documenteriez-vous l'architecture de ce lab pour un rapport d'audit ?
Un rapport d'audit du mini-lab comprend : (1) un schéma réseau avec toutes les IPs, les bridges, les VMs et les flux autorisés/bloqués ; (2) un inventaire des VMs avec OS, version, services en écoute (sortie de netstat -tlnp) ; (3) la liste des mesures de sécurité appliquées avec preuves (captures d'écran, sorties de commandes) ; (4) les règles de firewall documentées (ufw status verbose) ; (5) le journal des événements liés aux accès et modifications ; (6) une évaluation des risques résiduels avec recommandations. Cette documentation constitue la base d'une politique de sécurité et permet la reproductibilité du lab.

Questions techniques (§8.1)
1. Expliquez la différence entre un hyperviseur de Type 1 et de Type 2. Donnez un exemple de chaque.
Un hyperviseur Type 1 (bare-metal) s'exécute directement sur le matériel physique sans OS hôte intermédiaire. Il a un accès direct aux ressources hardware, ce qui lui confère de meilleures performances et une meilleure isolation. Exemples : VMware ESXi, Proxmox VE, Hyper-V Server, XenServer. Un hyperviseur Type 2 (hosted) s'exécute au-dessus d'un OS hôte comme une application. Il dépend de l'OS pour accéder au matériel, ce qui introduit une couche supplémentaire de latence. Exemples : VMware Workstation, VirtualBox, Parallels. Le Type 1 est utilisé en production pour ses performances et sa sécurité (pas d'OS hôte compromettable), le Type 2 est réservé aux postes de développement et aux tests.
2. Qu'est-ce qu'un vSwitch (ESXi) ? Comment se compare-t-il à un bridge Linux (Proxmox) ?
Un vSwitch ESXi est un commutateur virtuel interne qui connecte les cartes réseau virtuelles (vNIC) des VMs aux interfaces physiques (NICs) de l'hôte. Il supporte des port groups (avec VLAN IDs), des politiques de sécurité (promiscuous, MAC changes, forged transmits), et des VLANs. Un bridge Linux (Proxmox) est l'équivalent natif du noyau Linux. vmbr0 connecte les VMs au réseau physique, vmbr1 sans uplink crée un réseau isolé. La différence principale : le bridge Linux est plus configurable (intégration iptables, Open vSwitch possible) et gratuit ; le vSwitch ESXi est plus simple mais ses fonctionnalités avancées (vSwitch distribué) nécessitent vCenter payant.
3. Quelle est la différence entre thin provisioning et thick provisioning ? Quel impact sur les performances et le stockage ?
Thin provisioning : l'espace disque est alloué dynamiquement à mesure que les données sont écrites. Un disque de 20 GB ne prend physiquement que l'espace réellement utilisé (ex: 3 GB au départ). Avantage : économie d'espace, déploiement rapide. Risque : sur-allocation possible si de nombreuses VMs remplissent leurs disques simultanément. Thick provisioning : l'espace total est réservé immédiatement dès la création. Un disque de 20 GB occupe 20 GB dès le départ. Avantage : performances I/O prévisibles, pas de risque de manque d'espace. Inconvénient : consommation immédiate de l'espace, déploiement plus lent. En cybersécurité, le format Thick Eager Zeroed (ESXi) est recommandé car il écrase l'espace avec des zéros, empêchant la récupération de données résiduelles d'anciens tenants.
4. Expliquez le concept de snapshot. Pourquoi ne remplace-t-il pas une sauvegarde ?
Un snapshot capture l'état d'une VM à un instant T : état du disque (fichier delta des modifications), état de la mémoire (optionnel), et configuration. Il permet un retour arrière rapide en cas d'erreur. Il ne remplace pas une sauvegarde car : (1) il est dépendant du disque original — si le datastore est corrompu ou le serveur tombe, le snapshot est inutilisable ; (2) les snapshots accumulés dégradent les performances (lecture dans des fichiers delta en chaîne) ; (3) ils sont stockés au même endroit que la VM, donc non protégés contre les pannes matérielles ou ransomwares. Une vraie sauvegarde (vzdump, Veeam) est une copie complète et indépendante, idéalement stockée sur un support distinct et hors site.
5. Quelle commande permet de lister les VMs sur ESXi en CLI ? Et sur Proxmox ?
Sur ESXi en SSH : vim-cmd vmsvc/getallvms (liste toutes les VMs avec leur ID, nom, datastore, OS) ou esxcli vm process list (VMs actuellement en cours d'exécution). Sur Proxmox : qm list (liste les VMs KVM avec leur statut, RAM, PID) ou pvesh get /nodes/<node>/qemu via l'API REST. Pour les conteneurs LXC : pct list. Ces commandes sont essentielles pour les scripts d'automatisation et les audits de l'inventaire des VMs.

Questions cybersécurité (§8.1)
6. Listez 5 mesures de durcissement que vous appliqueriez sur un ESXi en production.
1) Désactiver SSH et ESXi Shell en dehors des interventions planifiées, avec activation du timeout automatique (UserVars.ESXiShellTimeOut = 600). 2) Créer des comptes nominatifs avec rôles limités (Read-Only, VM User) et supprimer le partage du mot de passe root. 3) Activer le Lockdown Mode (avec vCenter) pour interdire toute connexion directe à l'hôte en dehors de vCenter. 4) Envoyer les logs vers un SIEM via Syslog.global.logHost — les logs locaux peuvent être supprimés par un attaquant. 5) Appliquer les politiques de sécurité des port groups (Promiscuous = Reject, MAC changes = Reject, Forged transmits = Reject) et configurer NTP pour la fiabilité des logs.
7. Expliquez le principe du moindre privilège. Comment le RBAC de Proxmox permet-il de l'appliquer ?
Le principe du moindre privilège stipule que tout utilisateur, processus ou système doit avoir uniquement les droits nécessaires à l'accomplissement de sa tâche, sans plus. Cela limite la propagation en cas de compromission d'un compte. Le RBAC Proxmox applique ce principe à plusieurs niveaux : attribution de rôles prédéfinis (PVEVMUser, PVEAuditor, PVEDatastoreUser) ou personnalisés ; restriction par chemin (/vms/100 = accès à une seule VM) ; gestion par groupes ; realms multiples (PAM pour les admins système, PVE pour les utilisateurs internes). Ainsi, un opérateur peut démarrer/arrêter ses VMs sans avoir accès aux configurations réseau ou aux autres VMs.
8. Quels sont les risques d'une interface d'administration exposée sur Internet ? Comment s'en protéger ?
Une interface admin (ESXi :443 ou Proxmox :8006) exposée sur Internet est ciblée par des bots de scan en permanence. Risques : brute-force des credentials, exploitation de CVE dans l'interface web (injections, path traversal), utilisation de credentials volés, et accès root donnant contrôle total de l'infrastructure. Protections : (1) ne jamais exposer directement — accès via VPN IPsec/WireGuard uniquement ; (2) firewall réseau bloquant ces ports depuis Internet ; (3) fail2ban sur l'hôte ; (4) authentification à deux facteurs (TOTP) nativement supporté par Proxmox ; (5) alertes sur connexions depuis nouvelles IP ; (6) maintenir les patchs de sécurité à jour.
9. Un conteneur LXC non privilégié est-il « suffisamment isolé » pour héberger un service critique ? Argumentez.
Un conteneur LXC non privilégié offre une isolation nettement meilleure qu'un LXC privilégié car les UIDs/GIDs sont remappés (l'UID 0 dans le conteneur ne correspond pas à root sur l'hôte). Cependant, il partage toujours le noyau Linux de l'hôte. Pour un service critique (base de données de production, service d'authentification, données médicales), je considère que ce n'est pas suffisant car : tout CVE dans le noyau peut permettre une évasion ; le noyau partagé signifie qu'une vulnérabilité affecte tous les conteneurs simultanément ; les primitives de sécurité (seccomp, AppArmor) réduisent la surface mais ne l'éliminent pas. Une VM KVM est recommandée pour les services critiques car elle virtualise un noyau complet indépendant.
10. Comment configureriez-vous la centralisation des logs d'un ESXi vers un SIEM ? Quel est l'intérêt ?
Configuration via l'interface ESXi : Manage → System → Advanced Settings → Syslog.global.logHost, valeur : udp://<IP_SIEM>:514 ou tcp://<IP_SIEM>:514 pour plus de fiabilité. Les logs vmkernel, hostd, auth, shell sont ainsi envoyés en temps réel. L'intérêt principal est la protection contre la suppression de preuves : un attaquant qui compromet l'hôte ne peut pas effacer les logs déjà envoyés au SIEM. Le SIEM peut alors corréler : tentatives de connexion SSH, modifications de configuration, arrêts/suppressions de VMs, et déclencher des alertes automatiques. En conformité PCI-DSS ou ISO 27001, la centralisation des logs est une exigence obligatoire.
